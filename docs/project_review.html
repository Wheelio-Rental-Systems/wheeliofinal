Wheelio: Project Review

A comprehensive technical analysis of the Wheelio Vehicle Rental System.

1. Technical Stack Overview

React / Next.js Spring Boot 3.2.3 Java 17 MongoDB Maven Jenkins Docker JWT Security

2. Database Status: Postgres or MongoDB?

Current Status: The project has transitioned to a MongoDB-first architecture.

MongoDB (Active): All core entities (Users, Bookings, Vehicles, DriverProfiles, DamageReports) are currently mapped to MongoDB collections using Spring Data MongoDB.
PostgreSQL (Legacy/Optional): While infrastructure (docker-compose) still contains Postgres definitions, the application code has been migrated to MongoDB. No active JPA/Entity mappings exist in the current primary backend.

3. CI/CD Architecture (Jenkins)

The project utilizes Jenkins for automated build and integration processes. The Jenkinsfile defines a multi-stage pipeline:

  • Checkout: Robust Git fetching with manual configuration to handle high-latency connections (increasing postBuffer).
  • Backend Build: Compiles the Spring Boot application using mvn clean install, skipping tests for speed during integration.
  • Frontend Build: Installs dependencies and builds the React application using npm run build.
  • Artifact Archiving: Stores the resulting .jar and frontend assets for deployment.

4. Core Project Components

A. Booking Life Cycle

A sophisticated flow from vehicle discovery to rental completion. Key features include:

  • Dynamic vehicle availability checks.
  • Integrated driver selection and assignment.
  • Real-time status updates (PENDING, CONFIRMED, COMPLETED).

B. Driver Ecosystem

A dedicated module for driver interactions:

  • Secure Login: Role-based access control with JWT authentication.
  • Dashboard: Real-time overview of assigned trips and earnings.
  • Documentation: Secure upload and persistence of driver credentials.

C. Damage Reporting System

An essential rental feature allowing drivers to report vehicle issues with:

  • Detailed descriptions and photo uploads.
  • Admin-initiated cost estimation and approval flow.
  • Integrated payment processing for repair costs.

D. Real-time Notifications

To enhance user and driver engagement, the system implements a real-time notification layer:

  • Live Alerts: Instant toast notifications for new bookings and status changes.
  • Notification Center: A dedicated UI component for managing unread alerts with persistence in MongoDB.
  • Polling Mechanism: Backend-to-Frontend communication ensures data stays fresh without page reloads.

5. Important Things to Explain

If explaining this project to a panel or client, focus on these unique architectural decisions and functional highlights:

  • Microservices Readiness: While the current core is a monolith, the structure in backend-microservices shows a clear path toward a distributed architecture, allowing independent scaling of services like payments or booking.
  • Stateless Security: Using JWT (JSON Web Tokens) ensures the system remains scalable and works seamlessly across multiple backend instances. Authentication is handled via a JwtRequestFilter.
  • Document-Oriented Data: The choice of MongoDB allows for flexible schema evolution. This is particularly useful for Damage Reports where data (descriptions, costs, photo URLs) can vary significantly.
  • Automated Resilience (Jenkins): The pipeline is configured to handle network-constrained environments by tuning Git post-buffers, ensuring reliable builds even on volatile connections.
  • End-to-End Payment Flow: The integration of Razorpay for damage report payments demonstrates a complete financial transaction lifecycle within the app.
